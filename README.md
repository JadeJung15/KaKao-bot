# 픽셀곰 운영봇

카카오톡 오픈채팅방 운영 보조용 픽셀곰 봇 서버와 판매용 웹/앱 서비스입니다.


## 현재 범위

- 채팅방 봇 웹훅: `POST /chat-event`
- 카카오 i 스킬 웹훅: `POST /skill`
- 상태 확인: `GET /status`, 개발자용 원본 `GET /health`
- 서버/앱 진단: `/health`에서 DB 상태, 서버 시간, 앱 최소/최신 버전, 업데이트 필요 여부 확인
- 브릿지 앱 진단: 홈에서 알림 권한, 서버 연결, 등록 방, 앱/서버 버전 상태 확인
- 구매자 콘솔 온보딩: 신청, 승인, 연결코드, 앱 설치 단계를 체크리스트로 표시
- 픽셀곰 기능 소개 홈페이지: `GET /`
- 회원가입/서비스 신청/로그인: `GET /signup`, `GET /apply`, `GET /login`
- 구매자 전용 가이드/콘솔: `GET /buyer-guide`, `GET /console`, `GET /my-rooms`, `GET /setup`, `GET /license`, `GET /command-store`, `POST /api/buyer/guide`, `POST /api/buyer/console`
- 운영자 전용 어드민: `GET /admin`, `/api/admin/*`
- 게임 기능: 방별 기능이 켜진 경우 채팅 미니게임 사용
- 저장소: 로컬 JSON 또는 PostgreSQL
- 2차 운영 기능: 메시지함, 입퇴장 상세, 관리자 전용 명령어
- 3차 포인트 기능: 포인트, 좋아요, 응원, 뽑기, 홀짝, 이체, 출석, 레벨, 순위
- 4차 입장 안내: 신규 입장 환영문, 재입장 경고/이력, 닉네임 변경 카드
- 5차 브릿지 진단: 원본 이벤트 로그, 고유값 후보 확인, 고유값 기반 회원 연결 준비
- 6차 자동추출: 브릿지 `eventType`, `targetName`, `fromName`, `toName`, `profileHash` 자동 인식
- 7차 임시 진단: `/최근이벤트`에 profileHash 기준 이전 닉네임 요약 표시
- 8차 복구 표시: 회원 테이블에 없는 이전 닉네임도 rawEvents의 같은 해시에서 복구 표시
- 9차 전체방 복구: 방 키가 달라져도 전체 원본 이벤트에서 같은 고유값의 이전 닉네임 표시
- 10차 고유값 타임라인: `/최근이벤트`가 같은 고유값의 닉변 전후 원본 이벤트를 함께 표시
- 11차 첫 채팅 재입장 안내: 입장 시스템 메시지가 없어도 같은 고유값의 단일 이전 닉네임을 첫 일반 채팅에 안내
- 12차 재입장 후보 안내: 같은 고유값에 여러 이전 닉네임이 묶이면 첫 일반 채팅에 후보로 안내
- 13차 닉네임 변경 공개 안내: 고유값 기반 첫 채팅 알림에 닉네임 변경 감지 문구를 함께 표시
- 14차 직전닉 우선 안내: 고유값 후보가 여러 개여도 공개 알림에는 같은 방의 직전 활동 닉네임을 우선 표시
- 15차 닉변 안내 재표시: 잘못된 후보 안내가 이미 나간 현재 닉네임도 직전닉 기준으로 1회 다시 안내
- 16차 실사용 문구 단순화: 참여자 공개 알림에는 닉네임 변경, 이전닉, 현재닉만 표시
- 17차 포인트 콘텐츠: 응원 카드, 뽑기, 홀짝 추가
- 18차 고유값 오탐 방지: 같은 `profileHash`가 이미 존재하는 다른 참여자에게 붙으면 닉변 안내를 중단
- 19차 방 분리 안전값: 브릿지는 기본적으로 실제 카카오 방명을 보내고, 방명이 잘못 잡히는 환경에서만 방별 고정값을 사용
- 20차 방 링크 기능 제거: `/건의방`, `/얼공방`, `/무한성`, `/링크등록` 명령어 제거
- 21차 `/공질` 기능 제거: 공식질문 양식 명령어와 사이트/도움말 노출 제거
- 22차 개인톡 기록 차단: 브릿지와 서버에서 비그룹 메시지를 저장 전에 무시하고, `미정/익명` 같은 예약 이름은 순위에서 제외
- 23차 등록방 보호: `https://open.kakao.com/o/gu25P5vi` roomId가 있는 메시지만 처리
- 24차 판매용 서비스: 회원 신청, 수동 입금 승인, 라이선스 키, 구독 만료 차단, 방별 기능 ON/OFF, 백업/복구
- 25차 구매자 전용 안내: 승인된 구매자만 설치/권한/문제 해결 가이드를 서버에서 조회
- 26차 판매 흐름 단순화: 계정 생성과 서비스 신청을 분리하고, 승인된 방마다 브릿지 앱 연결코드를 발급
- 27차 운영자/구매자 분리: `/admin`은 판매자 전용, 구매자는 `/console`, `/my-rooms`, `/setup`, `/license`만 사용
- 28차 인증 확장 준비: Supabase 이메일 인증과 카카오 로그인 연결 준비, Kanana AI 활용 후보 로드맵 추가
- 29차 가입/요금 개선: 회원가입 닉네임, 브랜드 이메일 인증 템플릿, 간단 서버 상태 화면, 추가 방 2,200원 요금 안내
- 30차 어드민 안정화: 등록방 삭제 API/UI, 로그인 세션 표시 보정, 관리자 화면 스크롤 안정화, 카카오 로그인 요청 scope 축소
- 31차 명령어 스토어: 대표 명령어 템플릿 카탈로그, 구매자 방 설치 API, 자체 전용 브릿지 엔진 앱 경쟁력 문구 추가
- 32차 UX 정리: 사이트 이미지 깨짐/중복 개선, 홈 대문 압축, 명령어 스토어 빠른 보기/편집 설치/즐겨찾기/장바구니/설치 명령어 관리 추가
- 33차 방별 운영 보강: 방 선택 삭제, 신청/결제 기록 삭제, 추가 방 승인 금액 반영, `/` 없는 커스텀 명령어, 명령어 스토어 더 보기 레이아웃 추가
- 34차 명령어 스토어 세트화: 운영/이벤트/상점/게임 세트 템플릿, 세트 Editor, 실사용 응답문구, 일괄 설치 API 보강
- 35차 방별 관리 안정화: 어드민 방 카드에 기능 ON/OFF, 라이선스, 브릿지, 구독, 설치 명령어 수 상태 배지 표시
- 36차 명령어 스토어 품질: 설치 전 카카오 응답 미리보기, 참여자용 빠른 필터, 설치된 명령어 검색/편집 불러오기
- 37차 구독/연결 안정화: 만료 7일/3일/1일 안내, 만료 후 명령어/브릿지 연결 차단, 앱 연결값 오류 안내 정리
- 38차 백업 안전장치: 백업 JSON 스키마 버전, 복구 dry-run 검증 API/UI, 복구 실패 원인 요약 추가
- 39차 배포 자동화: 배포 전 필수 검사 스크립트, 배포 후 주요 URL smoke 스크립트, 롤백 기준 문서화
- 40차 Android 릴리즈 안정화: 1.0.19(20) 비공개 테스트 빌드 기준, 앱/서버 버전 호환 안내, AAB SHA256 출력 스크립트 추가
- 41차 운영 장애 대응: 서버/DB/권한/구독 만료 메시지 표준화, 진단 요약, 공지/업데이트 장애 대응 카드 추가
- 42차 설정 소유권 보정: 일반 채팅 이벤트가 웹 어드민 방별 기능 설정을 덮어쓰지 않도록 서버 기준 저장 흐름으로 정리
- 43차 명령어 팩 정리: 중복 템플릿을 대표 명령어 중심으로 줄이고 10종 명령어 팩을 추가
- 44차 카톡 설치 복사: 장바구니에서 `/명령어설치 ...` 문구를 복사하고 채팅방에서 `/설치확인`으로 설치
- 45차 스토어 UX 효율화: 팩으로 시작/명령어 찾기/내 장바구니 3모드와 sticky 카톡 설치 요약으로 재구성
- 46차 신규 구매자 UX: 회원가입, 서비스 신청, 승인 확인, 앱 연결, 첫 명령어 설치를 같은 5단계 흐름으로 안내하고 콘솔/스토어의 다음 행동 CTA를 정리


아직 등록되지 않은 `/` 명령어에는 안내 메시지만 반환합니다.
`/?`, `/도움말`, `/help`는 참여자와 관리자에게 서로 다른 목록을 보여줍니다. 참여자 도움말에는 관리자 전용 명령어를 노출하지 않습니다.

## 명령어 스토어

`GET /command-store`와 `GET /api/command-templates`는 구매자가 명령어를 쇼핑하듯 고를 수 있는 대표 템플릿 카탈로그입니다. 현재 카탈로그는 중복 조합을 제거한 대표 명령어, 추천 세트, 기본 운영, 참여자용, 관리자용, 게임 연결, 상점/아이템, 이벤트/시즌, 커뮤니티, AI 운영도우미 후보로 분류합니다.

구매자 콘솔에서 승인된 방이 있으면 설치 가능한 문구형/게임 연결/세트 템플릿을 `POST /api/buyer/command-templates/install`로 방별 커스텀 명령어에 담습니다. 고정 명령어와 관리자 전용 명령어는 예약어로 보호하며, 채팅방 응답에는 요금, 라이선스 키, 운영자 토큰 같은 민감 정보를 노출하지 않습니다.

명령어 스토어 화면은 `팩으로 시작`, `명령어 찾기`, `내 장바구니` 3모드로 나눕니다. 기본 흐름은 팩/대표 명령어를 고르고 장바구니에서 카톡 설치 명령어를 복사하는 방식이며, 사이트 직접 설치는 고급 영역으로 유지합니다. 구매자는 설치 전 명령어와 응답 문구를 직접 편집할 수 있고, 세트 템플릿은 여러 명령어와 응답을 한 번에 편집한 뒤 일괄 설치합니다. 설치된 커스텀 명령어는 `POST /api/buyer/custom-commands`와 `POST /api/buyer/custom-commands/delete`로 조회/삭제합니다. 명령어 팩은 `GET /api/command-packs`와 `POST /api/buyer/command-packs/apply`로 방별 복수 장착과 제거를 처리합니다.

장바구니에 담은 항목은 `카톡 설치 명령어 복사`로 `/명령어설치 pk.001 no.100 102 st.001` 형식의 문구를 만들 수 있습니다. 방 관리자가 카톡방에 붙여넣으면 봇이 설치 예정 항목을 먼저 보여주고, `/설치확인 코드` 입력 후 실제 설치합니다. 직접 만든 커스텀 명령어와 충돌하는 항목은 덮어쓰지 않고 건너뜁니다.

커스텀 명령어는 방별로 분리됩니다. `/공지`, `공지`, `!공지`, `.공지`는 서로 다른 명령어로 저장되고 실행됩니다. 예를 들어 `/공지=오늘 공지는 여기입니다.`, `공지=맛있어`, `!공지=차가워`를 같은 방에 동시에 둘 수 있습니다.

메시지함은 `@닉네임` 멘션을 받은 사람에게 읽지 않은 메시지를 보관하고, 대상자가 채팅을 치면 `/메시지` 안내를 보냅니다.
신규 입장에는 운영 규칙 환영문을 반환하고, 2회 이상 재입장에는 강퇴 이력과 입퇴장 히스토리를 함께 표시합니다.
닉네임 변경 감지 시 `이전닉 -> 새닉` 형식과 최초닉/변경닉 히스토리를 반환합니다.
`/최근이벤트`와 `/원본로그`는 관리자 전용입니다. 카톡 브릿지가 전달한 이벤트 요약과 `targetUserId/profileHash` 같은 고유값 후보를 확인할 때 사용합니다. 같은 `profileHash`로 닉네임이 바뀐 기록이 있으면 전체 방의 회원 테이블과 원본 이벤트를 함께 보고, 닉변 전후 원본 이벤트와 `회원이력`을 같은 고유값 기준으로 표시합니다. 입장 시스템 메시지가 브릿지에 잡히지 않아도 같은 고유값 사용자가 새 닉네임으로 첫 일반 채팅을 치면, 참여자 공개 알림에는 같은 방의 직전 활동 닉네임을 이전닉으로 우선 표시하고 현재닉만 함께 보여줍니다. 단, 같은 `profileHash`가 이미 존재하는 다른 참여자에게 번갈아 붙으면 해당 고유값은 공유/오탐으로 보고 공개 닉변 안내에 사용하지 않습니다.
`/고유값초기화 닉네임`은 이름이 섞인 회원의 고유값 연결과 자동 닉변 후보를 관리자 권한으로 정리합니다.
`/프로필등록`, `/프로필삭제`, `/별명등록`, `/별명삭제`, `/입퇴장상세`는 관리자 전용입니다.
`/관리자재설정`은 방 관리자 목록을 입력한 닉네임 목록으로 교체하고, `/관리자초기화`는 방 관리자 목록만 비웁니다. 환경변수 `ADMIN_NAMES` 관리자는 유지됩니다.
`/포인트지급`, `/포인트차감`, `/포인트설정`도 관리자 전용입니다.
초기 관리자는 아무나 등록할 수 없습니다. Vercel/로컬 환경변수 `ADMIN_NAMES`에 관리자 닉네임을 먼저 지정한 뒤 `/관리자등록 닉네임`으로 방 관리자 목록에 추가합니다.

## 포인트 규칙

`KakaoTalk_DB.txt`에서 확인한 운영형 포인트 규칙을 반영했습니다.

- 일반 채팅 1건: 포인트 +2, 경험치 +1
- 출석 1일 1회: 포인트 +100, 경험치 +10
- 레벨업: 포인트 +100
- 좋아요: `/좋아요 닉네임 수량`, 1~999개, 하트 1개당 포인트 4 차감
- 응원 카드: `/응원 닉네임 메시지`, 50포인트 차감, 대상자 하트 +1
- 뽑기: `/뽑기`, 100포인트 차감, 공개 확률에 따라 0~500포인트 획득
- 뽑기목록: `/뽑기목록`으로 확률과 보상 확인
- 홀짝: `/홀 금액` 또는 `/짝 금액`, 입력한 포인트를 베팅하고 맞히면 x2 지급
- 포인트 이체: `/이체 닉네임 포인트`, 수수료 10%, 최소 수수료 1포인트
- 순위: 출석, 포인트, 좋아요, 레벨, 오늘 채팅, 이번 주 채팅
- 상점/가방: 방별 상품은 가상 포인트로만 구매·사용·선물 가능
- 제외: 실제 금전성 아이템, 환전, 현금성 보상, 외부 결제형 아이템

참여자 도움말에는 구독 정보, 라이선스 키, 요금, roomId 같은 운영 민감 정보를 노출하지 않습니다. 해당 정보는 운영자 어드민/구매자 콘솔에서만 확인합니다.

## 배포

Vercel 배포 기준입니다.

```text
/
/health
/chat-event
/skill
```

`vercel.json`에서 `/`는 픽셀곰 기능 소개 홈페이지로, `/health`는 서버 상태 확인 API로 연결됩니다. `/status`는 구매자·운영자가 보기 쉬운 정상/주의/오류 카드 화면이고, `/health?versionCode=앱버전코드`는 브릿지 앱 업데이트 필요 여부까지 확인합니다.

도메인 역할:

```text
https://pixgom.com/              픽셀곰 소개 사이트 대표 도메인
https://ka-kao-bot.vercel.app/   Vercel 기본 앱 도메인
https://pixgom.com/chat-event   안드로이드 브릿지 기본 API 주소
```

운영 브릿지는 혼동을 줄이기 위해 `https://pixgom.com/chat-event`를 기본 주소로 사용합니다. Vercel 기본 도메인은 장애 진단용 보조 주소로만 둡니다.

배포 전 기준:

1. `npm.cmd run check:deploy`가 통과해야 합니다.
2. 체크 스크립트는 `node --check`, `git diff --check`, `npm run test:local`을 순서대로 실행합니다.
3. 실패하면 커밋/푸시/배포를 중단하고 같은 Phase에서 원인을 수정합니다.

배포 후 기준:

1. Vercel production 배포 후 `npm.cmd run smoke:prod`를 실행합니다.
2. `/health`, `/command-store`, `/admin`, `/console` 응답과 핵심 문구를 확인합니다.
3. `/health`의 `version`, `dbStatus.ok`, 신규 feature 플래그가 기대값인지 확인합니다.

Android 비공개 테스트 AAB 기준:

1. Play Console 업로드 트랙은 항상 비공개 테스트(Closed testing)입니다.
2. `npm.cmd run android:bundle`로 release AAB를 생성하고 SHA256, 경로, versionName/versionCode를 확인합니다.
3. Android 빌드 산출물과 업로드 키는 `.vercelignore`로 Vercel 배포에서 제외합니다.
4. 빌드만 검증하고 Play Console 약관 동의, 권한 승인, 실제 업로드는 운영자 계정에서 직접 확인합니다.

롤백 기준:

1. 배포 후 `/health`가 실패하거나 `dbStatus.ok`가 false이면 즉시 직전 정상 Vercel deployment로 롤백합니다.
2. `/command-store`, `/admin`, `/console` 중 하나라도 빈 화면이거나 핵심 문구가 없으면 새 배포를 사용하지 않습니다.
3. 롤백 후 `/updates`에는 장애 원인과 복구 시각을 별도 카드로 기록합니다.

7. 여러 방은 웹 관리 콘솔에서 등록하고, 앱에는 봇폰이 실제로 들어가 있는 대표 방을 먼저 설정합니다.

## 회원가입과 라이선스 흐름

1. `/signup`에서 닉네임, 이메일, 비밀번호, 비밀번호 확인을 입력하고 서비스 이용약관/개인정보처리방침에 동의해 계정을 만듭니다.
2. Supabase 이메일 인증이 켜져 있으면 인증 메일의 `이메일 인증하기` 버튼을 눌러 가입을 완료합니다.
3. `/apply`에서 방 이름, 오픈채팅 링크, 관리자 닉네임을 입력해 서비스를 신청합니다.
4. 관리자가 입금 확인 후 운영자 전용 `/admin`에서 신청을 승인합니다.
5. 승인 시 방별 라이선스 키와 앱 연결코드가 생성되고 30일 구독이 시작됩니다.
6. 구매자는 `/console`, `/my-rooms`, `/setup`, `/license`에서 자기 방의 연결코드와 라이선스만 확인합니다.
7. 구독 만료 후에는 해당 방의 서버 응답이 차단됩니다.

요금은 기본 방 1개 30일 5,500원입니다. 같은 계정에서 방을 추가 신청하면 추가 방마다 30일 2,200원을 결제 대상으로 표시합니다.

## 운영자 어드민과 구매자 콘솔

`/admin`은 판매자/운영자 전용입니다. 전체 구매자, 전체 방, 구독 만료일, 라이선스 발급/차단, 백업/복구를 관리하므로 `OWNER_ADMIN_EMAILS`에 등록된 운영자 이메일 로그인으로 보호합니다.

`ADMIN_CONSOLE_TOKEN`은 자동화나 긴급 복구용 비상 키로만 남겨둘 수 있습니다. 일반 운영 화면에서는 토큰을 직접 입력하지 않습니다.

구매자는 `/admin`을 사용하지 않습니다. 구매자 화면은 다음처럼 분리합니다.

```text
/login       구매자 로그인
/forgot-password 비밀번호 재설정 메일 요청
/reset-password 새 비밀번호 저장
/console     구매자 요약 콘솔
/my-rooms    승인된 내 방과 연결코드
/setup       앱 설치와 권한 안내
/license     라이선스/요금/만료 안내
```

초기 MVP에서는 로컬 계정과 구매자 세션 토큰을 같이 지원합니다. Supabase 환경변수를 설정하면 이메일/비밀번호 인증, 비밀번호 재설정, 카카오 로그인 버튼이 활성화됩니다.

## Supabase/Kakao 로그인 설정

1. Supabase 프로젝트를 만들고 `SUPABASE_URL`, `SUPABASE_ANON_KEY`를 Vercel 환경변수로 설정합니다.
2. Supabase Auth URL Configuration에 `https://pixgom.com/login`, `https://pixgom.com/console`, `https://pixgom.com/signup`, `https://pixgom.com/apply`, `https://pixgom.com/reset-password`를 허용 리다이렉트 URL로 추가합니다.
3. Kakao Developers에서 웹 플랫폼 도메인 `https://pixgom.com`을 등록합니다.
4. Kakao Login Redirect URI에는 `https://pixgom.com/api/auth/kakao/callback`을 등록합니다. Supabase 기본 OAuth를 같이 유지할 때만 Supabase 콜백 URL `https://프로젝트.ref.supabase.co/auth/v1/callback`도 추가합니다.
5. Kakao REST API 키를 Vercel `KAKAO_REST_API_KEY`에 넣습니다. 카카오 앱에서 Client Secret을 켠 경우 `KAKAO_CLIENT_SECRET`도 설정합니다.
6. Kakao Login 일반 설정에서 OpenID Connect를 `ON`으로 켜고, 동의항목은 `profile_nickname`, `profile_image`만 허용합니다. `account_email` 권한이 없는 앱은 이메일 범위를 요청하지 않아야 `KOE205`를 피할 수 있습니다.
7. Vercel의 `SUPABASE_KAKAO_ENABLED=true`, `KAKAO_OIDC_ENABLED=true`를 설정합니다. OIDC 환경변수가 없으면 Supabase 기본 OAuth로 폴백합니다.
8. 설정 후 `/api/auth/config`가 `mode: "supabase"`, `kakaoEnabled: true`, `kakaoMode: "oidc"`를 반환하는지 확인합니다.

이메일 인증 메일은 Supabase Auth Email Templates의 Confirm signup 템플릿에 `docs/supabase-email-confirmation-template.html` 내용을 붙여 넣어 사용합니다. 템플릿에는 픽셀곰 로고, `이메일 인증하기` 버튼, 24시간 유효 안내, 발신 전용 안내가 포함되어 있습니다.

## Kanana AI 활용 후보

카카오가 Hugging Face에 공개한 Kanana 모델은 한국어 운영 문구 생성, 공지/규칙 초안, 커스텀 명령어 답변 추천, 문의 답변 초안에 활용할 수 있는 후보입니다.

현 단계에서는 바로 서버에 붙이지 않습니다. 모델별 라이선스, 상업 이용 조건, 추론 비용, 개인정보 입력 차단 정책을 확인한 뒤 별도 AI 기능으로 분리합니다.

우선 후보:

- `kakaocorp/kanana-1.5-2.1b-instruct-2505`: Apache-2.0 표기, 가벼운 테스트 후보
- `kakaocorp/kanana-2-30b-a3b-instruct-2601`: Kanana License 표기, 성능 후보지만 상업 조건 검토 필요
- `kakaocorp/kanana-nano-2.1b-embedding`: 검색/FAQ/문서 유사도 후보

MessengerBot 스크립트는 과거 호환용으로만 남겨둡니다. 신규 판매/테스트 기준은 픽셀곰 브릿지 앱입니다.

현재 운영 URL:

```js
const BOT_SERVER = "https://pixgom.com/chat-event";
```

방 분리 설정:

```js
const ROOM_NAME_OVERRIDE = "";
const ROOM_LINK = "https://open.kakao.com/o/gu25P5vi";
const ROOM_ID = "gu25P5vi";
const ALLOWED_ROOMS = [];
const ALLOWED_ROOM_IDS = ["gu25P5vi"];
```

`ROOM_NAME_OVERRIDE`는 기본값을 비워둡니다. 이렇게 해야 카카오톡이 전달한 실제 방 이름으로 `state.rooms[roomKey]`가 나뉘어 서로 다른 채팅방 기록이 분리됩니다.
메신저봇 환경에서 방 이름이 내 닉네임처럼 잘못 들어오는 경우에만, 해당 방 스크립트에 방별 고정값을 넣습니다.
`ALLOWED_ROOMS`는 빈 배열이면 모든 방 허용이고, 운영 방만 제한하려면 전송될 방 이름을 배열에 넣습니다.
`ROOM_ID`는 오픈채팅 링크의 `/o/` 뒤 ID를 넣습니다. 서버는 등록된 `ROOM_ID`가 없거나 다른 메시지를 저장 전에 무시합니다.
등록된 `ROOM_ID`가 있으면 일부 메신저봇 환경에서 `isGroupChat/rawIsGroupChat`이 false로 잡혀도 운영방 메시지로 처리합니다.
개인톡이나 1:1 대화처럼 등록된 `ROOM_ID`가 없는 메시지는 서버에서도 저장 전에 무시합니다.

브릿지는 카카오 알림에 포함된 메시지만 서버로 전송합니다. 기본 운영에서는 화면 감지/접근성 권한을 사용하지 않습니다.
신규 판매 기준은 타 앱 방별 스크립트가 아니라 픽셀곰이 자체 개발한 전용 브릿지 엔진 앱입니다. 앱, 서버, 웹 콘솔의 데이터 계약을 직접 통제하므로 방별 설정, 라이선스, 구독 만료, 기능 토글을 일관된 방식으로 관리할 수 있습니다.

## 픽셀곰 전용 브릿지 방향

현재 기본 브릿지는 Android 알림 접근 권한을 사용합니다. 카카오톡이 알림으로 보내지 않는 입장/퇴장 시스템 문구는 기본 방식만으로는 감지할 수 없습니다. 그래서 운영 기준은 방장봇 입장확인 문구, 첫 채팅 보정, 고유값 오탐 방지, 최근 이벤트 진단을 조합합니다.

| 문제 | 현재 메신저봇 방식 | 전용 브릿지 목표 |
|---|---|---|
| 방 이름 | 테스트 위치에 따라 내 닉네임처럼 잘못 들어올 수 있음 | 방별 고정/정확한 방 식별 |
| 입장/퇴장 | 알림으로 오지 않는 시스템 문구는 수신 불가 | 방장봇 문구와 첫 채팅 보정 |
| 닉네임 변경 | 첫 채팅 기준 추정이라 오탐 가능 | 이벤트/고유값 기반 추적 |
| 고유값 | `profileHash`가 여러 사람에게 섞일 수 있음 | 더 안정적인 사용자 식별값 확보 |
| 운영 안정성 | 메신저봇 앱 설정/방별 스크립트 의존 | 픽셀곰 전용 앱/웹 콘솔로 통제 |

## 구현 원칙

- 실제 결제/현금성 아이템/유료 게임 연동은 별도 승인 전까지 활성화하지 않습니다.
- 게임 기능은 우선 채팅 미니게임과 향후 픽셀곰 RPG 연동 준비로만 관리합니다.
- 먼저 방 운영에 필요한 프로필, 별명, 링크, 입퇴장, 닉변 기록을 안정화합니다.
- 저장이 필요한 기능은 로컬 JSON과 PostgreSQL 저장소를 모두 지원합니다.
- 운영 데이터 변경 명령어는 관리자 전용으로 분리합니다.
